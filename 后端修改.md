## 前端调试指引：确认 WebSocket 是否真的推送了 execute_command 的交互事件

### 0. 这份指引的“前因后果”（为什么要这样查）
你遇到的现象是：
1) 模型输出了 `execute_command`，例如启动 `npx expo start --lan --port 8082`  
2) 但前端时间线/日志在模型回复后就“停住了”，看不到 `command_start / command_output`  
3) 后端子进程其实已经进入交互分支，例如 Expo 的 `? Use port 8083 instead? (Y/n)`  

这类问题有两个常见原因：
1) **AI/后端并没有真正把输入写入 stdin**（也就是交互 prompt detector 没命中，或写入链路被阻塞）  
2) **AI/后端已经产生了 command 事件，但前端没有收到或被过滤掉了**  

你之前反馈“前后端打印都没看到任何东西”，并且前端展示在某些轮次会突然停止，这更像是第 2 类问题：**WebSocket 增量补发存在漏事件**。  

本项目里 WebSocket 增量原本使用的是 `lastStepId` 作为游标，但 `execute_command` 的 `command_start/command_output/command_end` 有可能与相邻事件复用/相近 step_id，导致前端只按 `stepId > lastStepId` 过滤时，漏掉同一步骤内的后续事件。  

因此我们需要在前端做两件事：
- 把 WebSocket 收到的每条消息都打印出来（证明到底有没有 `command_output`）
- 断线重连时把游标从只用 `lastStepId` 扩展到 **`lastEventSeq`**（避免同 step 内漏事件）

后端已经在每条 WebSocket 消息中附带 `eventSeq`（字段名：`eventSeq`）。请前端利用它。

---
### 1. 在 WebSocket `onmessage` 里逐条打印消息（第一证据）
在前端订阅 `WebSocket /conversations/ws/{conversationId}/{runId}` 的代码里，确保你的 `onmessage` 回调里逐条 `console.log`，并且更新游标。

推荐写成这样（你可以直接照抄，路径/变量名按你的前端工程做轻微调整）：

```ts
let lastStepId = 0;
let lastEventSeq: number | null = null;

function connectWs(url: string) {
  const ws = new WebSocket(url);

  ws.onmessage = (ev) => {
    const msg = JSON.parse(ev.data);
    console.log("[ws-msg]", {
      type: msg.type,
      stepId: msg.stepId,
      eventSeq: msg.eventSeq,
      title: msg.title,
      detail: msg.detail,
    });

    // 只要拿到 stepId/eventSeq，就推进游标
    if (typeof msg.stepId === "number") {
      lastStepId = msg.stepId;
    }
    if (typeof msg.eventSeq === "number") {
      lastEventSeq = msg.eventSeq;
    }
  };

  return ws;
}
```

你要重点关注的事件类型：
- `type === "command_start"`：表明后端已经开始执行命令
- `type === "command_output"`：表明 stdout/stderr（以及 stdin 写入后对应的输出）有增量
- `type === "command_end"`：命令结束
- （用于上下文核对）`type === "tool_call"` / `type === "tool_result"`

---
### 2. 断线重连时同时带上 `lastEventSeq`（避免“同 step 内事件漏推送”）
后端现在会在每条消息中附带 `eventSeq`。  
请你前端维护两个游标：
- `lastStepId = msg.stepId`
- `lastEventSeq = msg.eventSeq`

重连时拼接 query 参数：
- `?lastStepId=<lastStepId>&lastEventSeq=<lastEventSeq>`

如果你暂时没有改动前端的重连逻辑，也至少请做到：**重连时不要只带 lastStepId**。否则仍可能漏掉 `command_output/stdin`。

后端兼容策略说明（你不需要理解细节，但知道它会影响你看到的结果）：
- 如果前端没有提供 `lastEventSeq`，后端会退回到更保守的策略（容易重复，但不应漏关键输出）
- 但为了让你更容易对比“是否真的推送过”，强烈建议你把 `lastEventSeq` 也传上

---
### 3. 触发场景的“验收标准”（你该看到什么）
你触发这种命令（端口冲突）：
- `npx expo start --lan --port 8082`

根据 Expo 的交互提示，理想情况下你应该看到前端消息序列大致包含：
1. `command_start`（detail/title 大概率包含执行的 command 字符串）
2. 随后至少一条 `command_output`，其中 `detail` 文本包含类似：
   - `Port 8082 is being used by another process`
   - `Use port 8083 instead?`
3. 若后端已经实现了自动 stdin，应继续出现更多 `command_output`（可能包含类似 `stdin` 写入相关内容），并且进程不会一直停在 `? Use port ... (Y/n)` 等输入等待上

你需要做的核对点是：  
- 如果你看到 **Expo 交互提示**的 `command_output`，但后面仍卡住，那么问题更可能在“写 stdin 没成功”
- 如果你从来没看到 Expo 交互提示对应的 `command_output`，那问题更可能是“前端漏事件/过滤问题”

---
### 4. 如果你愿意把问题定位到更细：请把最后一段 `ws-msg` 发我
每次卡住时，把最后 20~50 条 `console.log("[ws-msg]", ...)` 的输出贴出来（至少包含字段：`type/stepId/eventSeq/title/detail`）。  
我可以据此判断是：
- 前端根本没收到 `command_output`
- 还是收到了但被你的 UI 逻辑过滤/未渲染
- 或者事件收到了但没有对应 stdin 写入事件
