## Expo Go：从初始页跳转到后端生成工程（`exp://...`）指引

### 1. 背景与目标
你现在的流程是：后端“直接生成另一个 Expo 工程并运行起来”，并能拿到可用的 `exp://...` URL；前端希望从初始页跳转到这个新工程，并在手机上的 `Expo Go` 里显示目标页面。

本文档用于前后端对齐实现方式与关键约束，默认你的“可用 URL”来源于 Expo 开发服务器（Metro / packager）在运行时打印出来的 `exp://` 链接。

### 2. 核心可行性与限制
1. `exp://...` 方式通常要求手机能够网络访问到“运行该新工程的那台机器”的 IP 和端口。
2. 若手机与后端机器不在同一局域网（Wi-Fi/有线网），或者中间有防火墙/端口拦截，则 `Expo Go` 可能无法拉取 JS bundle，表现为加载失败/空白页。
3. `exp://...` URL 通常只对“当前运行中的开发服务器进程”有效；当后端重新启动/进程退出，URL 可能失效。

因此：
- 在“同一局域网 + 端口可达”的情况下，这种方式通常可工作。
- 若要跨网络/生产环境更稳妥的做法通常是 `EAS Update` / `u.expo.dev`（本文先不展开，因为你当前已经有 `exp://...` URL）。

### 3. 约定总体流程
1. 前端发起“生成并跳转”请求给后端。
2. 后端在后台生成 Expo 工程、安装依赖、启动该工程的 Expo 开发服务器。
3. 后端从启动日志中提取 `exp://...` URL，并返回给前端（同时返回一个 `requestId` 用于后续查询/轮询状态）。
4. 前端拿到 `exp://...` 后，调用 `Linking.openURL()`，让 `Expo Go` 打开该体验/包。

### 4. 后端需要做什么（生成 + 启动 + 返回 URL）
#### 4.1 工程生成部分（概念要求）
你的“生成出来的 Expo 工程”需要满足：
- 这是一个标准的 Expo 工程（`package.json` + `app.json/app.config.*` + `src/` 或 expo 路由结构等）。
- 能够在不依赖用户手动操作的情况下完成依赖安装和启动。

建议你在生成时确保：
- 目标页面/路由确实可在该工程的启动后被命中（例如通过固定入口 `index` 或 query 参数控制页面）。
- 生成工程的配置不会和并发请求互相覆盖（并发时使用独立目录/独立端口）。

#### 4.2 启动开发服务器并确保手机可访问
关键点是让手机能访问到 packager/bundler 的地址与端口。

你需要做两件事：
1. 让 Metro 服务监听 `0.0.0.0`（这样能被局域网访问）。
2. 确保你启动命令使用了 `--lan`（或等价方式）并能输出 `exp://...` 链接。

一个常见的启动方式形如（具体命令参数可按你项目的 Expo 版本微调）：

```bash
# 在生成工程目录内执行
# --lan：让 expo start 输出局域网可用的链接（通常包含 exp://）
# --host 0.0.0.0：监听所有网卡，保证局域网设备可访问
# --port <PORT>：选择端口，建议你为每个 request 分配独立端口以支持并发
npx expo start --lan --host 0.0.0.0 --port <PORT>
```

#### 4.3 提取 `exp://...` 并返回给前端
启动后，Expo CLI 通常会在 stdout 里打印类似 `exp://<ip>:<port>` 的 URL。

你后端需要：
- 捕获启动进程的 stdout
- 用正则提取 `exp://` 开头的第一个（或最合适的）URL
- 返回给前端，并给出 `requestId` 与可能的过期策略

示例（伪代码 / Node.js 思路，需你按实际后端语言改造）：

```ts
// 示例：Node.js 后端启动子进程并提取 exp:// URL
// 注意：这是思路示例，不保证与你当前代码直接可编译。
import { spawn } from "child_process";

function extractExpUrlFromChunk(chunk: string): string | null {
  // 匹配 exp://<ip>:<port> 的核心部分
  // 说明：
  // - <ip> 这里用简化的 IPv4 正则：1~3 位数字点号重复
  // - <port> 用 1~5 位数字
  // - 你也可以根据实际输出更精确地匹配路径/参数
  const match = chunk.match(/exp:\/\/\d{1,3}(\.\d{1,3}){3}:\d{1,5}/);
  return match ? match[0] : null;
}

export async function startGeneratedExpoAndGetUrl(options: {
  projectDir: string;
  port: number;
}): Promise<{ expUrl: string; requestId: string }> {
  const requestId = crypto.randomUUID();

  const child = spawn(
    "npx",
    ["expo", "start", "--lan", "--host", "0.0.0.0", "--port", String(options.port)],
    { cwd: options.projectDir }
  );

  let expUrl: string | null = null;

  // 监听 stdout
  child.stdout.on("data", (data) => {
    const chunk = data.toString("utf8");

    // 尝试从每一段输出中提取 URL
    const maybe = extractExpUrlFromChunk(chunk);
    if (maybe && !expUrl) {
      expUrl = maybe;
    }
  });

  child.stderr.on("data", (data) => {
    // 将错误日志记录下来，便于定位启动失败
    console.error("[expo start stderr]", data.toString("utf8"));
  });

  // 需要等待直到 expUrl 出现（或超时）
  const timeoutMs = 120_000;
  const start = Date.now();
  while (!expUrl) {
    if (Date.now() - start > timeoutMs) {
      // 启动超时通常意味着网络/依赖/端口冲突等问题
      child.kill();
      throw new Error(`Timeout waiting for exp:// url, requestId=${requestId}`);
    }
    await new Promise((r) => setTimeout(r, 500));
  }

  return { expUrl, requestId };
}
```

#### 4.4 URL 与进程生命周期管理（强烈建议）
建议你在后端维护一张表（内存/数据库都可以）：
- `requestId`
- `projectDir`
- `port`
- `pid`（expo start 进程 pid）
- `expUrl`
- `status`（starting/running/failed/expired）
- `expiresAt`（可选）

原因：
- 前端拿到 `expUrl` 之后期望页面能持续打开；进程提前退出会导致后续打开失败。
- 并发请求时需要隔离端口与目录。
- 为避免资源泄漏，你需要回收过期工程（停止子进程、删除临时目录或至少释放端口）。

### 5. 前端需要做什么（调用后端 + Linking 打开）
#### 5.1 接口约定（建议）
你可以把后端抽象为两步：
1. 触发生成并返回可打开链接
2. 如需轮询，提供查询接口

示例接口（按需调整路径/字段名）：
- `POST /api/expo-experiences/generate`
  - body：`{ initialParams: {...} }` 或 `{ prompt: "...", ... }`
  - response：
    - `requestId: string`
    - `expUrl: string`（例如 `exp://192.168.1.10:19000`）
    - `expiresAt?: number`（毫秒时间戳，可选）
    - `status: "ready"`
- `GET /api/expo-experiences/:requestId`
  - response：`{ status, expUrl?, error? }`

#### 5.2 前端跳转代码（TypeScript 示例）
```ts
import * as Linking from "expo-linking";

async function openGeneratedExpoExperience(expUrl: string) {
  try {
    // canOpenURL：在某些平台/情况下可以提前判断链接是否可处理
    // 注意：如果 expUrl 格式不对或环境限制，canOpenURL 可能返回 false
    const supported = await Linking.canOpenURL(expUrl);
    if (!supported) {
      // 这里建议做 UI 提示：无法打开目标体验
      // 以及提供退回按钮
      throw new Error("Expo Go 无法处理该 expUrl");
    }

    // 触发 Expo Go 打开目标体验
    // 说明：exp:// 链接会让 Expo Go 尝试拉取对应开发服务器的 bundle
    await Linking.openURL(expUrl);
  } catch (e: any) {
    // 常见问题：
    // - 手机无法访问后端机器 IP/端口（同网/防火墙）
    // - 后端进程尚未完全启动（URL 未就绪）
    // - URL 过期（进程已退出）
    // 建议把 expUrl 与错误信息一起上报日志，方便定位
    console.error("Failed to open exp experience:", { expUrl, error: e?.message ?? String(e) });
    throw e;
  }
}

// 使用示例：先向后端请求 expUrl，再跳转
export async function handleJump() {
  // 1) 请求后端生成并获取 expUrl
  const res = await fetch("/api/expo-experiences/generate", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ /* 你的生成参数 */ }),
  });

  if (!res.ok) {
    throw new Error(`generate failed: ${res.status}`);
  }

  const data = await res.json();

  // 2) 取出 expUrl，并打开
  // 注意：如果你后端可能返回 status=starting，需要轮询直到 ready
  if (!data.expUrl) {
    throw new Error(`No expUrl returned. status=${data.status}`);
  }

  await openGeneratedExpoExperience(data.expUrl);
}
```

### 6. 交互时序建议
为了避免前端拿到 URL 时后端还没启动完毕，你可以让后端只在 `expUrl` 准备好后才返回 `status=ready`。

建议时序（概念）：
1. 前端点击按钮
2. 前端 `POST generate`
3. 后端启动 expo start
4. 后端提取 `expUrl` 且确认 ready
5. 返回 `expUrl`
6. 前端 `Linking.openURL(expUrl)`

### 7. 常见故障排查清单
1. `Linking.openURL` 失败或立即无反应
   - 检查 `expUrl` 格式是否正确（确保是 `exp://` 开头，且包含 ip:port）
   - 检查手机端 Expo Go 是否正常可用
2. 打开了但一直转圈/加载失败/空白
   - 检查手机是否和后端机器在同一局域网
   - 检查后端机器端口是否开放（Windows 防火墙需要放行 `port`）
   - 检查 expo start 是否真的在监听并已完成启动
3. 只能某些设备打开
   - 可能是 IP 路由/网段不通、防火墙策略不同
4. 过一会儿后失败
   - 说明后端子进程可能超时被回收了；需要调整 `expiresAt` 或回收策略

### 8. 安全注意（建议写进实现里）
- 后端开放的 Metro 端口如果暴露到公网，风险较高；建议仅在局域网使用，或者通过内网访问控制/Tunnel/鉴权进行隔离。
- 生成的临时工程目录建议限制访问、避免注入未清洗的数据进入 shell 命令。
